# Chapter 09: useState를 이용한 반응형 상태 제어와 불변성 (Reactive State and Immutability)

## 📖 핵심 개념
React는 단방향 데이터 흐름을 가지며, 상태 변경을 감지하여 UI를 업데이트합니다. 이때 객체나 배열과 같은 참조 타입의 상태를 변경할 때는 불변성(Immutability)을 지켜야만 반응성이 보장됩니다.

*   **단방향 데이터 바인딩과 불변성**:
    *   상태 데이터는 단방향으로만 흐르며, 참조형 데이터를 업데이트할 때는 기존 데이터를 직접 수정하는 대신 새로운 복사본을 생성해 업데이트해야 합니다. (예: `[용어 사전](../../../../GLOSSARY.md#immutability)` 참고)
    *   **Friendly_Tutor의 친절한 비유**: "전광판의 숫자를 펜으로 직접 덧칠해 수정(직접 변경)하는 대신, 새 전광판으로 완전히 교체(불변성 유지 상태 갱신)해야만 화면 갱신 센서가 동작하여 바뀐 화면을 보여주는 것과 같습니다."
*   **참조 주소의 불일치와 리액트 렌더링 생략(Skip)**:
    *   **Senior_Practitioner의 실무 팁**: "React의 state 배열에 `push()`를 사용해 직접 수정한 뒤 `setState(state)`를 호출하는 실수를 자주 범합니다. 이 경우 참조 메모리 주소가 동일하기 때문에 React는 상태 변화가 일어나지 않았다고 판단하여 화면 렌더링을 완전히 생략(Skip)해 버리는 치명적 버그가 발생합니다. 따라서 항상 스프레드 연산자(`...`)나 `.filter()`, `.map()` 등을 사용해 새로운 참조 주소를 가진 배열로 상태를 변경해야 합니다."
*   **Object.is 비교 메커니즘과 Asynchronous Batching**:
    *   **PhD_Book_Author의 학술 스펙**: "React는 `Object.is` 비교 알고리즘을 사용하여 이전 상태와 다음 상태의 동일성 여부를 평가합니다. 두 객체의 참조 주소가 같으면 업데이트가 무시됩니다. 또한 React의 상태 업데이트 큐(Queue)는 성능 최적화를 위해 동기적으로 즉시 렌더링하는 대신, 이벤트 핸들러 실행이 완료될 때까지 보류했다가 일괄 처리(Asynchronous Batching)하여 단 한 번의 화면 갱신만 수행하는 방식으로 메모리 및 연산 낭비를 통제합니다."

## 🧪 실습 미션
*   **미션 파일**: `mission.ts`
*   **요구 사항**:
    1.  `updateStateHelper<T>(currentState: T[], action: { type: 'add' | 'remove'; payload: T }): T[]`:
        - 원본 배열 `currentState`를 직접 변경(Mutation)하지 않고 불변성을 유지하며 작동해야 합니다.
        - `action.type`이 `'add'`인 경우, `payload`를 배열 끝에 추가한 새 배열을 반환합니다.
        - `action.type`이 `'remove'`인 경우, `payload`와 일치하는 요소를 제거한 새 배열을 반환합니다.
        - 어떤 경우든 반환되는 배열의 메모리 참조 주소는 `currentState`와 완전히 달라야 합니다.

## 💡 힌트 및 트러블슈팅
*   배열 요소를 추가할 때는 스프레드 연산자(`...`)를 이용해 `[...currentState, action.payload]`와 같이 새 배열을 생성하고, 삭제할 때는 `filter` 메서드를 사용하여 조건을 충족하는 요소만 남긴 새 배열을 리턴하여 원본 참조와 분리하세요.
