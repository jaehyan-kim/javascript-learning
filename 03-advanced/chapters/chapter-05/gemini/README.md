# Chapter 05: React 성능 최적화: 대용량 리스트 렌더링 최적화: 가상 스크롤 구현

## 📖 핵심 개념
대용량 데이터를 리스트 형식으로 렌더링할 때, 전체 아이템을 한 번에 DOM에 그리지 않고 현재 사용자의 화면 뷰포트에 보이는 영역의 아이템만 제한적으로 DOM에 올려 브라우저 메모리 부하와 렌더링 지연을 극복하는 기법(가상 리스트)입니다.

*   **렌더링 가상화 (Virtual List)와 DOM 노드 수 고정**:
    *   현재 화면상에 표출되어야 하는 유효한 범위의 요소만 선택적으로 렌더링하여 메모리를 절약하는 구조적 패턴입니다. (자세한 내용은 [용어 사전](../../../../GLOSSARY.md#virtual-scrolling) 참고)
    *   **Friendly_Tutor의 친절한 비유**: "10,000페이지짜리 파노라마 두루마리 그림을 미술관 벽에 전부 펼쳐 놓는 대신, 관람객이 서 있는 뷰파인더 화면 창(컨테이너) 크기에 딱 맞는 3~4개의 그림 칸만 액자에 붙여 두고 관람객이 움직일 때마다 그림 내용만 빠르게 갈아 끼우는 것과 같습니다."
*   **DOM Node Bloat 현상과 대규모 피드 장애**:
    *   수천 수만 개의 DOM 노드가 메모리에 한 번에 적재되어 브라우저의 힙 메모리가 가득 차고 GC(Garbage Collector)의 지연으로 브라우저 탭이 얼어버리는(Crash) 현상입니다.
    *   **Senior_Practitioner의 실무 팁**: "10,000개의 대용량 SNS 피드 데이터를 받아서 아무 가상화 없이 화면에 전부 `map`으로 뿌렸다가 브라우저 힙 메모리가 급증하여 모바일 브라우저가 강제 종료되거나 스크롤 반응 속도가 극도로 느려지는 장애가 흔히 발생합니다. 가상화 기법을 도입하여 DOM 트리에 부착되는 노드 개수를 수십 개 수준으로 고정해야 실무급 애플리케이션으로 동작할 수 있습니다."
*   **레이아웃 스래싱 (Layout Thrashing)과 뷰포트 기하학**:
    *   자바스크립트에서 DOM의 기하학적 속성(예: `scrollTop`, `offsetHeight`)을 빠르게 읽고 쓰는 과정이 반복되어 브라우저가 불필요한 동기식 레이아웃 계산을 강제 실행하게 되는 병목 현상입니다.
    *   **PhD_Book_Author의 학술 스펙**: "가상 스크롤을 구현하기 위해선 현재 스크롤 위치(`scrollTop`), 컨테이너의 물리적 높이(`containerHeight`), 개별 아이템 높이(`itemHeight`)를 이용해 연산해야 합니다. 스크롤 이벤트 수신 시 매번 동기적으로 DOM 값을 읽기보다 렉시컬 상태에 뷰포트 지형 기하학(Geometry) 상태를 담아두고 계산해야 레이아웃 스래싱을 방지할 수 있으며, 이에 맞추어 화면 바깥에 위치해야 할 여백(`offsetTop`)을 CSS 변환(Transform)이나 패딩으로 정확히 밀어주어 스크롤 바의 높이와 시각적 스크롤 싱크를 완벽히 일치시켜야 합니다."

## 🧪 실습 미션
*   **미션 파일**: `mission.ts`
*   **요구 사항**:
    1.  `getVisibleRange(scrollTop: number, containerHeight: number, itemHeight: number, totalCount: number): { startIndex: number; endIndex: number; offsetTop: number }`: 스크롤 높이에 맞춰 화면에 표출되어야 하는 가상 리스트 노드의 시작/끝 인덱스 및 상단 여백을 계산하여 출력하는 가상 스크롤 엔진을 구현하십시오.
        - `startIndex`: 현재 스크롤 위치에서 화면 상단 바깥으로 벗어난 아이템들을 제외하고, 뷰포트에 처음으로 나타나야 하는 아이템의 인덱스입니다. (0 이상이어야 함)
        - `endIndex`: 뷰포트 하단을 포함하여 화면 내에 렌더링되어야 하는 마지막 아이템의 인덱스입니다. (스크롤 시 여백 방지를 위해 뷰포트 외곽에 일부 여유분(overscan)을 두도록 계산되며, inclusive이고 `totalCount - 1` 이하로 제한됨)
        - `offsetTop`: 시작 아이템 이전까지의 영역 높이를 계산하여, 스크롤 내에서 렌더링되고 있는 아이템들이 올바른 위치에 자리 잡을 수 있도록 상단 여백을 계산해 반환해야 합니다.

## 💡 힌트 및 트러블슈팅
*   `startIndex`는 `scrollTop / itemHeight`의 내림 연산을 통해 구합니다.
*   화면에 표시될 수 있는 최대 아이템 개수(올림 연산)를 계산한 뒤, `startIndex`에 더하여 `endIndex`의 범위를 산정합니다. `totalCount`가 0인 엣지 케이스를 처리하여 음수 인덱스가 나오지 않도록 주의하세요.
*   `offsetTop`은 `startIndex * itemHeight`을 통해 이전 보이지 않는 아이템들이 차지하던 수직 공간 크기를 구하면 됩니다.
