# Chapter 03: 커스텀 상태 엔진 직접 구현

## 📖 핵심 개념
Flux 패턴의 단방향 데이터 흐름을 기반으로, 클로저 환경에 스토어 상태를 격리하고 상태 변화 시에만 구독자들에게 알림을 발송하는 경량 상태 관리 스토어입니다.

*   **렉시컬 스코프와 클로저**:
    *   함수 정의 시점의 환경을 기억하여 외부에서 접근할 수 없는 렉시컬 스코프 변수를 캡처하는 메커니즘입니다. (자세한 내용은 [용어 사전](../../../../GLOSSARY.md#closure) 참고)
    *   **Friendly_Tutor의 친절한 비유**: "구독 신청을 한 독자들(컴포넌트 리스너)의 주소록을 금고에 보관하고, 새로운 잡지(업데이트된 상태)가 발간될 때마다 누락 없이 잡지를 배달해 주는 자동 물류 본부와 같습니다."
*   **Object.is 비교와 렌더링 성능 최적화**:
    *   두 값이 물리적으로 동일한지 판별하는 엄격한 비교 함수입니다.
    *   **Senior_Practitioner의 실무 팁**: "상태 스토어를 설계할 때 값이 변경되지 않았음에도 `setState` 호출 시 얕은 비교 검증을 누락하면, 매번 무조건적인 구독 트리거를 발행하여 리액트 컴포넌트 전체가 불필요하게 리렌더링되는 성능 폭탄을 유발합니다. 반드시 `Object.is` 또는 얕은 비교 가드를 구현하여 상태가 변경되었을 때만 알림이 전파되도록 최적화해야 합니다."
*   **단방향 데이터 흐름 (Unidirectional Data Flow)**:
    *   상태의 변경은 오직 한 방향(Action/SetState -> Store -> View)으로만 흘러야 한다는 아키텍처 규칙입니다.
    *   **PhD_Book_Author의 학술 스펙**: "클로저를 활용해 상태 변수를 외부에서 직접 수정(Mutation)할 수 없도록 철저히 격리하며, 상태 변경은 오직 `setState`라는 단일 진입점만 통과하도록 차단합니다. 이를 통해 상태 변경의 예측 가능성을 보장하고 렉시컬 스코프 내의 상태 변경이 트리거되어 스토어의 구독 목록(Listeners)을 순차적으로 깨우는 순수 함수형 상태 구독 루프를 수립합니다."

## 🧪 실습 미션
*   **미션 파일**: `mission.ts`
*   **요구 사항**:
    1.  `createCustomStore<T>(initialState: T): Store<T>`: getState, setState, subscribe 메커니즘을 구비한 경량 스토어 코어 엔진을 구현하십시오.
        - `getState()`: 현재 상태 값을 즉시 반환합니다.
        - `setState(nextState)`: 새로운 상태를 인자로 받거나 상태를 변형하는 함수를 전달받아 상태를 업데이트합니다. 이때 객체 형태인 경우 기존 상태와 병합(Merge)되어야 합니다.
        - `setState` 호출 시 병합 후 각 키를 `Object.is`로 비교하는 얕은 비교를 수행하여 새로운 상태가 기존 상태와 다른 경우에만 구독된 리스너들을 실행해야 합니다.
        - `subscribe(listener)`: 리스너 함수를 구독 대기열에 등록하고, 호출 시 구독을 안전하게 해제(Unsubscribe)할 수 있는 고차 함수를 반환해야 합니다.

## 💡 힌트 및 트러블슈팅
*   구독자 리스너들을 중복 없이 안전하게 관리하기 위해 `Set<() => void>` 자료구조를 내부적으로 사용하면 중복 구독을 간편하게 방지할 수 있습니다.
*   `setState`를 사용할 때 들어오는 인자가 함수 형태(`(state: T) => Partial<T>`)인지 값 형태(`Partial<T>`)인지 타입을 체크하여 처리해 주어야 유연한 업데이트가 가능합니다.
