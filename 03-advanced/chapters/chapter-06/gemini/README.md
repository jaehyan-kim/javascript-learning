# Chapter 06: 동적 스레드 병렬 처리: Web Workers를 이용한 CPU 무거운 연산 분산 (Dynamic Thread Parallel Processing: Web Workers)

## 📖 핵심 개념
싱글 스레드 기반의 자바스크립트 엔진에서 무거운 CPU 연산이 메인 스레드를 점유하는 현상을 방지하고, 이를 별도의 백그라운드 스레드로 분산하여 애플리케이션의 반응성(UI 렌더링 및 인터랙션)을 극대화하는 멀티스레딩 기법입니다.

*   **웹 워커 (Web Worker)**:
    *   브라우저의 메인 스레드와 병렬로 동작하는 백그라운드 스레드를 생성하여 스크립트를 실행할 수 있게 해주는 HTML 표준 API입니다. (자세한 내용은 [용어 사전](../../../../GLOSSARY.md#web-worker) 참고)
    *   **Friendly_Tutor의 친절한 비유**: "식당에서 혼자 카운터 계산(UI 응대)과 주방 요리(무거운 연산)를 다 처리하다가 손님이 대기하는 시간이 길어져 컴플레인이 발생하는 상황이 싱글 스레드의 블로킹 현상입니다. 이때 주방 전담 요리사(Web Worker)를 고용하여 요리를 위임하고, 메인 스레드는 카운터에서 손님 응대(인터랙션)에 집중하며 요리가 완성될 때마다 전달받는 구조와 같습니다."
*   **postMessage / onmessage**:
    *   메인 스레드와 워커 스레드 간에 데이터를 주고받기 위한 비동기 메시지 전달 시스템입니다. 데이터는 메모리 주소를 공유하지 않고, 복사본을 만드는 안전한 통신 방식으로 전달됩니다.
    *   **Senior_Practitioner의 실무 팁**: "워커 스레드는 UI 스레드와 완전히 격리되어 있기 때문에 내부에서 `window`나 `document` 같은 DOM API에 접근하려 하면 즉시 `ReferenceError`가 발생하며 크래시가 일어납니다. 또한 메시지를 보낼 때 사용되는 **구조화된 복제(Structured Clone)** 메커니즘은 함수(`Function`), DOM 노드 등 복제가 불가능한 대상을 전송할 수 없으므로 데이터를 객체 형태로 직렬화하여 전달할 때 데이터 구조와 전송 오버헤드에 극도로 주의해야 합니다."
*   **이벤트 루프와 메시지 큐 연동**:
    *   **PhD_Book_Author의 학술 스펙**: "HTML Living Standard 명세에 따르면, 각 Web Worker는 메인 스레드와는 별개의 독립적인 실행 컨텍스트(Global Scope)와 자신만의 독립적인 [이벤트 루프](../../../../GLOSSARY.md#event-loop) 및 태스크 큐(Task Queue)를 갖습니다. 메인 스레드에서 `postMessage`를 호출하면 워커 스레드의 이벤트 루프가 관리하는 태스크 큐에 메시지 수신 이벤트 콜백이 푸시되어 비동기적으로 처리되며, 이 구조는 스레드 간 데이터 레이스(Data Race)와 데드락(Deadlock)을 방지하는 공유 메모리 배제형 액터 모델(Actor Model) 구조를 실현합니다."

## 🧪 실습 미션
*   **미션 파일**: `mission.ts`
*   **요구 사항**:
    1.  `createMockWorkerEnv(workerScript: string)`:
        - 메인 스레드와 워커 간의 메시지 송수신 및 콜백 시뮬레이션을 구현하여 비동기 병렬 연산 정합성을 검증하는 시뮬레이터를 구현합니다. (이 미션은 실제 OS 스레드나 진짜 병렬 실행이 아니라 학습용 단일 스레드 시뮬레이션입니다.)
        - 워커 내부 스크립트 실행 중 `window` 또는 `document`와 같은 DOM API에 접근하려 할 때 런타임에 `ReferenceError`를 발생시키며 크래시가 나도록 고안해야 합니다.
        - `postMessage` 호출 시 전송 불가 데이터(함수 등)에 대해 `DataCloneError`를 검출해 내도록 `structuredClone` 기반의 복제 예외를 모사해야 합니다.
        - `terminate()` 메소드를 제공하여 워커 실행을 즉시 중단하고 전송 대기 중(미처리)인 메시지를 포함해 이후 모든 메시지 전달을 차단해야 합니다.

## 💡 힌트 및 트러블슈팅
*   Node.js의 `node:vm` 모듈을 사용하면 전역 스코프가 격리된 샌드박스 실행 환경을 만들 수 있습니다. 샌드박스 내에 `window`와 `document`를 아예 선언하지 않음으로써 브라우저 API 무단 접근으로 인한 오류를 재현해 보세요.
*   `postMessage`를 통해 비동기로 처리되도록 모사해야 하므로 JavaScript의 `setTimeout`을 통해 마이크로태스크/매크로태스크 타이밍을 맞춰 전송 콜백이 호출되도록 구조화하세요.
