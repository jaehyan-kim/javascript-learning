# Chapter 08: React Server Components (RSC) 렌더링 동작 원리 및 Payload 분석 (RSC Payload Parsing)

## 📖 핵심 개념
React Server Components(RSC) 아키텍처에서 서버 컴포넌트가 실행된 결과를 직렬화하여 클라이언트로 스트리밍 전송할 때 사용하는 특수 데이터 포맷인 RSC Payload의 구조를 분석하고, 클라이언트 컴포넌트 참조 정보 및 서버 DOM 노드 트리 관계를 해석하여 렌더링 수명 주기를 이해하는 원리입니다.

*   **리액트 서버 컴포넌트 페이로드 (RSC Payload)**:
    *   서버 컴포넌트가 렌더링한 가상 DOM 트리 구조와 클라이언트 컴포넌트 참조 정보(Client Reference)를 직렬화한 텍스트 기반 데이터 포맷입니다. (자세한 내용은 [용어 사전](../../../../GLOSSARY.md#rsc-payload) 참고)
    *   **Friendly_Tutor의 친절한 비유**: "RSC Payload는 완제품 요리 가구(HTML)가 아니라 이케아의 가구 조립 설명서 및 부품 도면 데이터(RSC Payload)를 스트리밍해 와서 조립하는 것과 같습니다. 브라우저는 이 부품 도면을 한 줄씩 읽어가며 가상 DOM 구조를 파악하고 조립을 시작합니다."
*   **클라이언트 컴포넌트 상태 보존**:
    *   RSC Payload는 HTML이 아닌 추상 데이터이므로 기존 클라이언트 컴포넌트의 React 상태(State)를 보존하면서 화면을 업데이트할 수 있습니다.
    *   **Senior_Practitioner의 실무 팁**: "RSC가 클라이언트 컴포넌트의 상태(State)를 보존하면서 화면을 업데이트할 수 있는 핵심 이점을 모르면, 탭 전환이나 필터 변경 시 전체 페이지를 리로드하거나 불필요한 하드 네비게이션을 구현하는 실수를 범하게 됩니다. RSC Payload 파싱 메커니즘을 이해하면 데이터를 동적으로 스트리밍받아 기존 텍스트 입력창에 타이핑 중이던 값(State)을 잃지 않고 화면만 부드럽게 교체해 주는 고급 아키텍처 설계를 수립할 수 있습니다."
*   **RSC 와이어 포맷의 직렬화와 하이드레이션**:
    *   **PhD_Book_Author의 학술 스펙**: "React Server Components 명세에 따르면, RSC 와이어 포맷(RSC Wire Format)은 JSON 객체를 확장한 행 단위 문자열 스트림 구조를 띱니다. 서버 컴포넌트 렌더러는 비동기 작업이 완료되는 대로 청크 단위로 텍스트를 출력하여 브라우저로 흘려보냅니다. 각 행은 `J0`, `M1` 등 고유한 ID 참조 기호를 가져 상호 의존 관계를 정의하며, 브라우저는 HTML 파서와 동시에 비동기적으로 이를 수신하여 [클라이언트 컴포넌트](../../../../GLOSSARY.md#server-client-components) 번들을 지연 로딩하고 [하이드레이션(Hydration)](../../../../GLOSSARY.md#hydration) 단계에서 기존 가상 DOM 트리와 안전하게 매칭시킵니다. *(참고: 이 미션에서 다루는 형식은 실제 RSC 런타임/번들러 경계를 완벽히 재현한 것이 아닌 학습을 위해 단순화된 독자적 텍스트 포맷입니다.)*"

## 🧪 실습 미션
*   **미션 파일**: `mission.ts`
*   **요구 사항**:
    1.  `parseRscPayload(payload: string)`:
        - 줄 단위로 직렬화된 RSC Payload 문자열을 해독하는 파서를 구현합니다.
        - 각 줄은 `ID:TYPE:JSON` 형식으로 이루어져 있으며, `CLIENT` 타입은 클라이언트 레퍼런스 정의를 의미하고 `SERVER` 타입은 가상 DOM 노드와 하위 자식 노드 정보를 의미합니다.
        - 자식 노드 배열 내에 존재하는 다른 노드의 ID 문자열(예: `"M1"`)을 감지하여 재귀적으로 실제 노드 객체 정보로 해독 및 결합하여 완전한 트리를 빌드해야 합니다.
        - 순환 참조(Circular Reference)가 있는 잘못된 페이로드 데이터가 입력되었을 경우 이를 검출하여 `Circular reference detected` 예외를 발생시키도록 예외 처리해야 합니다.
        - 파싱 중 잘못된 행 형식을 만나면 `Invalid RSC line format`, 잘못된 JSON 형식을 만나면 `Invalid JSON` 예외를 발생시키도록 방어적 파싱 로직을 구현해야 합니다.

## 💡 힌트 및 트러블슈팅
*   먼저 전체 페이로드 문자열을 줄 단위로 분리한 후, 맵(Map) 객체를 생성하여 `ID`를 키로 하고 파싱된 임시 노드 정보를 값으로 설정하여 1차 정리하세요.
*   그 다음, 다른 노드에 의해 참조되지 않은 노드들이 최상위 루트 노드가 됩니다.
*   루트 노드들을 기점으로 재귀 함수(Recursive DFS)를 돌며 하위 자식 배열 내의 ID 문자열을 맵에서 조회한 노드 객체로 깊은 복사하여 교체 처리해 줌으로써 완성된 트리를 완성할 수 있습니다.
*   재귀 탐색 시 이미 방문한 노드 ID 목록을 셋(Set) 등으로 기억해 두어 동일한 경로에서 다시 마주치면 순환 참조 예외를 발생시키도록 하세요.
*   `ID:TYPE:JSON` 형식이 아닌 줄이 입력되면 예외를 던지고, `JSON.parse`가 실패할 때의 예외를 잡아 방어 처리하세요.
